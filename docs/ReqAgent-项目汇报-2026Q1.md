# ReqAgent 项目汇报

> 2026 Q1 · 智能需求分析平台

---

## 一、项目概述

ReqAgent 是一个 **AI 驱动的需求分析工作台**，是多 Agent 编排平台的第一个 Agent。

用户在 Web 界面中用自然语言描述需求，Agent 通过多轮对话完成需求理解、追问和补全，最终产出符合规范的需求文档。

```
用户输入需求 → 智能追问补全关键信息
             → 检索领域知识与历史参考
             → 多轮对话逐步明确需求细节
             → 生成符合规范的需求文档
```

### 3.4 Workspace 设计（Phase 2 规划）

当前架构设计中 Agent 为无状态对话模式。Phase 2 将引入 **Workspace** —— 每个会话拥有独立的虚拟文件系统，Agent 可以在其中读写文件、保存产物。

**候选方案对比：**

| 维度 | Vercel VM | E2B Sandbox |
|------|-----------|-------------|
| 隔离级别 | 函数级（serverless） | 完整微型虚拟机 |
| 文件系统 | 临时目录，函数存活期有效 | 持久化文件系统，跨请求存活 |
| 启动延迟 | ~0ms（热启动） | ~150ms（冷启动） |
| 存活时间 | 随函数生命周期（最长 5min） | 可配置（最长 24h） |
| 安全性 | 沙箱隔离，无 root | 完整 Linux VM，root 隔离 |
| 成本 | 含在部署费用中 | 按时长计费 |
| 适用场景 | 轻量文件操作 | 代码执行、复杂文件处理 |

**策略**：Phase 2 先用轻量方案满足"Agent 读写文件"的基本需求。后续需要代码执行能力时，引入 E2B 作为隔离沙箱。

## 四、风险分析

### 4.1 开发周期风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| Workspace 和外部服务集成周期不确定 | 产品化时间线延后 | 先用最简方案，不追求完美隔离 |
| 上游框架版本迭代快 | 升级维护成本 | 锁定大版本号，关键差异已记录 |
| 团队带宽有限 | 并行推进困难 | 严格优先级排序，各阶段串行推进 |

### 4.2 模型成本风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 多阶段流程每次消耗 3-4 次模型调用 | Token 成本线性增长 | 默认使用高性价比模型；阶段可选择性跳过 |
| 长文档处理消耗大量 Token | 单次请求成本高 | 结构化输出减少无效内容，节省 30-50% |
| 代理端点额外加价 | 实际成本高于直连 | 评估直连可行性；协商代理定价 |

### 4.3 国产模型替换风险

这是**最关键的技术风险**。当前架构基于 OpenAI 兼容协议，理论上可切换到任何兼容端点，但实际效果取决于模型能力：

| 能力维度 | 当前模型表现 | 国产模型预期风险 |
|----------|------------|----------------|
| 结构化输出 | 稳定，格式遵从度高 | **中风险** — 部分模型格式遵从度不够 |
| 中文需求理解 | 优秀 | **低风险** — 国产模型中文能力普遍较强 |
| 工具调用 | 稳定，原生支持 | **高风险** — 部分模型实现不完整 |
| 推理能力 | 支持 | **中风险** — 能力参差不齐 |
| 长上下文 | 128K | **中风险** — 部分模型窗口较小 |

**缓解策略：**

```
                        渐进式验证路径

  简单工具测试 ──→ 单阶段验证 ──→ 全流程接入 ──→ 生产部署
       │               │              │             │
       ▼               ▼              ▼             ▼
   知识检索工具     需求解析阶段    三阶段串联     稳定运行
   兼容性确认      输出质量评估    端到端验证     监控告警
```

- **协议层兼容**：只要端点兼容标准协议即可切换
- **降级方案**：工具调用不可靠时退化为提示词驱动模式
- **混合部署**：关键阶段用强模型，辅助阶段用低成本模型

## 五、Feature 规划

### 5.1 近期目标：Demo 核心能力

| Feature | 说明 | 价值 |
|---------|------|------|
| 多轮对话需求文档生成 | 通过多轮对话追问补全信息，产出符合规范的需求规格说明书 | 替代手工编写初稿，保证文档规范性 |
| 文件解析 | 支持上传 Word/PDF 等文档，Agent 提取内容作为需求输入 | 兼容现有工作流，降低使用门槛 |
| 领域知识检索 | 分析过程中自动检索领域知识，辅助需求完整性检查 | 减少遗漏，提升需求质量 |
| 产物文件管理 | 产物面板展示生成的文件，支持查看和下载 | 结果可交付 |

### 5.2 中期目标：产品化基础

| Feature | 说明 |
|---------|------|
| Workspace | 每个会话拥有独立虚拟文件系统 |
| 外部服务集成 | 接入文档解析、网页抓取等外部能力 |
| 领域知识热插拔 | 不同项目组加载不同知识包（PRD 模板、合规要求、术语表） |
| 需求迭代 | 用户反馈后增量更新文档，非全部重新生成 |

### 5.3 远期愿景：多 Agent 编排平台

当前 ReqAgent 验证了单 Agent 的技术可行性。未来将扩展为多 Agent 协作平台：

```
                         ┌───────────────┐
                         │  编排调度中心   │
                         │  Orchestrator  │
                         └───────┬───────┘
                                 │
                  任务分发 & Handoff
                                 │
          ┌──────────────────────┼──────────────────────┐
          │                      │                      │
   ┌──────▼───────┐    ┌────────▼────────┐    ┌────────▼────────┐
   │  ReqAgent    │    │   ModelAgent    │    │   CodeAgent     │
   │  需求分析     │    │   领域建模      │    │   代码生成       │
   │              │    │                │    │                 │
   │  → 需求文档  │    │  → ER 图       │    │  → 项目脚手架    │
   │  → 原型草图  │    │  → API 设计     │    │  → 单元测试      │
   └──────┬───────┘    └────────┬────────┘    └────────┬────────┘
          │                     │                      │
          ▼                     ▼                      ▼
     需求产物 ───────────→ 设计产物 ────────────→ 代码产物
     (下游消费)            (下游消费)             (可交付)
```

**核心设计**：每个 Agent 的产出物直接作为下一个 Agent 的输入，形成从需求到代码的自动化链路。新增 Agent 只需定义职责和工具，无需改动底层架构。

## 六、总结

### 当前进展

- 整体架构设计完成，关键技术路径验证通过
- 模型接入层支持灵活切换，为后续国产模型替换预留空间

### 下一步

- Demo 核心能力实现：多轮对话需求文档生成 + 文件解析
- 启动 Phase 2 Workspace 设计与实现
- 国产模型兼容性验证

### 核心观点

> **框架是管道，知识是水。** Agent 的上限不取决于技术架构，取决于它能调用多少领域知识。技术架构已经验证可行，下一步的关键投入方向是：**业务知识的结构化沉淀**。
