# ReqAgent — 智能需求分析平台

## 项目汇报

---

## 一、我们要解决什么问题

当前需求分析过程中存在三个核心痛点：

**效率低** — PM 手工编写 PRD 耗时长，从需求沟通到文档输出通常需要 1-2 周。需求变更时文档修改成本高，经常出现版本不一致的问题。

**质量不稳定** — PRD 质量高度依赖个人经验。新人容易遗漏非功能需求、合规要求、边界条件。评审时反复返工，同样的问题在不同项目中重复出现。

**知识没有沉淀** — 历史项目的需求分析经验散落在个人电脑里，没有形成可复用的组织资产。新项目不能有效参考过往经验。

---

## 二、我们的方案

构建一个 **AI 驱动的需求分析助手**（ReqAgent），作为多 Agent 编排平台的第一个 Agent。

### 它能做什么

用户在 Web 界面上输入需求描述，Agent 自动完成：

```
用户输入需求 → Agent 智能追问补全信息
             → 自动检索领域知识和历史参考
             → 输出结构化 User Story（含优先级和验收标准）
             → 生成完整的 PRD 文档（Markdown 格式）
```

### 核心能力

| 能力 | 说明 |
|------|------|
| 智能追问 | 自动识别需求描述中的缺失信息，主动向用户追问关键问题 |
| 知识复用 | 检索领域知识模式，参考类似功能的拆分方式和常见问题 |
| 标准化输出 | 按结构化模板生成 User Story 和需求文档 |
| 领域适配 | 不同项目组可配置自己的业务知识（后续 Skill 热插拔） |

## 三、产品形态

### 用户交互流程

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   ① 用户输入        ② Agent 分析         ③ 产出物       │
│                                                         │
│   一句话描述  ──→   智能追问      ──→   User Story 列表  │
│   需求文档    ──→   领域知识检索  ──→   PRD 文档         │
│   补充细节    ──→   需求拆解      ──→   优先级矩阵       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 界面设计

采用 monochrome 设计系统，gallery-first 开发模式：

- **左侧**：对话区域，用户与 Agent 自然语言交互
- **右侧**：产物文件列表，展示生成的 User Story 和需求文档
- **实时状态**：Agent 执行过程中显示当前活动（推理中、调用工具、生成回复），让用户清楚了解进度

## 四、Skill 热插拔 — 适配不同业务线（Phase 4）

### 设计理念

Agent 本身是通用的需求分析引擎。不同项目组通过加载不同的 **Skill**（领域知识包）来适配自己的业务场景。

```
同一个 Agent 引擎
    │
    ├── 加载"银行零售 Skill"   → 变成银行零售需求分析专家
    ├── 加载"证券交易 Skill"   → 变成证券交易需求分析专家
    └── 加载"风控平台 Skill"   → 变成风控平台需求分析专家
```

### Skill 包含什么

| 组成 | 说明 | 示例 |
|------|------|------|
| 领域指令 | 告诉 Agent 该领域的分析重点 | "银行零售业务需关注渠道适配" |
| PRD 模板 | 该业务线的标准文档模板 | 公司内部 PRD 模板 |
| 合规知识 | 相关的监管和合规要求 | 个保法、反洗钱条款 |
| 历史参考 | 过往项目的 PRD 摘要 | 手机银行项目需求参考 |
| 领域术语 | 专业术语标准定义 | 头寸、T+0、KYC 等 |

### 谁来维护 Skill

**业务团队**直接维护，不需要写代码。Skill 就是一组 Markdown 文件，按约定格式整理即可。技术团队只负责 Agent 引擎和平台能力。

## 五、技术方案概要

### 架构选型

| 层级 | 选择 | 理由 |
|------|------|------|
| 整体框架 | 全栈 TypeScript (Next.js 15) | 一个项目、一个语言、部署简单 |
| AI 运行时 | Vercel AI SDK v6 | 行业主流，月下载 2000 万+ |
| 前端 UI | assistant-ui | 成熟的 AI 对话组件库，可组合原语 |
| 设计系统 | CSS Modules monochrome | gallery-first 开发，组件一致性高 |

### 架构简图

```
┌──────────────────────────────────────────────┐
│  Web 界面                                     │
│  AI 对话组件 + 产物文件列表                     │
└──────────────┬───────────────────────────────┘
               │
┌──────────────▼───────────────────────────────┐
│  Agent 运行时                                 │
│  需求解析 → 知识检索 → 需求拆解 → 文档生成     │
│  + 领域知识注入（后续 Skill 热插拔）           │
└──────────────┬───────────────────────────────┘
               │
┌──────────────▼───────────────────────────────┐
│  工具层                                       │
│  知识库检索 · 结构化输出 · 文档生成             │
│  后续：MCP Server · Workspace FS              │
└──────────────────────────────────────────────┘
```

## 六、多 Agent 平台演进路线

ReqAgent 是平台的第一个 Agent。架构设计已考虑后续扩展。

```
Phase 1（已完成）      Phase 2            Phase 3          Phase 4-5
基础对话 + 工具  →    Workspace FS    →  MCP 集成      →  Skill + 多 Agent
Tool calling          文件读写           外部服务          知识热插拔
Gallery 组件          Thread 隔离        协议标准化        Handoff 编排
```

## 八、需要的支持

### 技术侧
- 全栈 TS 开发 1-2 人
- OpenAI 兼容 API Key

### 业务侧（最关键）
- **PM 团队**：提供公司标准 PRD 模板和撰写规范
- **合规团队**：提供金融监管要求摘要
- **各项目组**：提供 2-3 份历史项目 PRD 作为参考
- **业务分析师**：提供领域术语标准定义

> 业务知识的质量直接决定 Agent 的产出质量。框架是管道，知识是水。

## 九、预期价值

| 维度 | 当前 | 使用 ReqAgent 后 |
|------|------|-----------------|
| PRD 编写周期 | 1-2 周 | 1-2 天（初稿） |
| 新人上手成本 | 需要资深 PM 带教 | Agent 内置标准流程和知识 |
| 知识复用 | 散落在个人电脑 | Skill 沉淀为组织资产 |
